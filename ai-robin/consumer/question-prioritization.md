# Question Prioritization

Methodology for ranking the must-ask questions Consumer surfaces to the
user. Used in Phase 3 after Gap Analysis identifies must-ask items.

The underlying principle: **Consumer has finite interaction budget. Every
question consumed is a question you can't ask later. Prioritize the
questions that, if wrong, cause the most damage downstream.**

---

## The three ranking dimensions

### 1. Blast radius

How much of the project downstream depends on this decision.

High blast radius:
- **Authentication approach** — affects auth pages, API middleware, DB
  schema (sessions / tokens tables), frontend state, deployment
- **Database engine** — affects ORM, schema syntax, query patterns, hosting
- **Framework (e.g., Next.js vs Remix)** — affects file structure,
  deployment model, every feature's code style
- **Deployment target** — affects build pipeline, runtime code (edge vs
  node), environment variables, scaling assumptions

Low blast radius:
- UI color scheme — localized to CSS
- Specific icon library — swappable
- Logging format — centralized in one module

**Rule**: high blast radius items rank higher. A wrong high-blast-radius
choice means a costly replan cycle or a degraded scope.

### 2. Reversibility

How expensive it is to change this decision later.

Irreversible (expensive to change):
- Database engine (data migration pain)
- Deployment platform (infrastructure relearning)
- Monorepo vs polyrepo
- Major framework (Next.js → Remix is a ground-up rewrite)
- License (hard to re-license third-party contributions)

Reversible (cheap to change):
- Styling library (swap CSS)
- Component library (replace components one by one)
- Logging library (wrapper + swap)
- Individual library choices within a framework

**Rule**: irreversible items rank higher. If Planning / Execute has to
change a reversible decision later, it's annoying; an irreversible one
forces project restart or degradation.

### 3. Ask-ability

How cheap and unambiguous it is to ask the user.

Highly ask-able:
- "Which deployment target?" with concrete options — user answers in
  one line
- "Postgres / SQLite / in-memory?" — multiple choice
- "Support multi-user or single-user?" — binary

Low ask-ability:
- "What's your preferred error handling philosophy?" — requires
  context the user doesn't have; user probably doesn't know what they
  prefer
- "How should we handle edge cases in search ranking?" — implementation
  detail
- "What's your testing strategy?" — too broad; user just wants the
  project to work

**Rule**: low ask-ability items should NOT be asked. They should be
proxy-decided or deferred to Planning. Asking them wastes budget and
annoys the user.

---

## Combined ranking

For each must-ask item, score it roughly:

- **High blast radius + irreversible + ask-able** → top priority. Ask
  first.
- **High blast radius + reversible + ask-able** → ask if budget permits.
- **Low blast radius + irreversible + ask-able** → ask if still have
  budget.
- **Anything with low ask-ability** → don't ask; proxy.

Within a tier, prefer items whose answer unlocks other questions. A
single good answer can clarify multiple gaps.

---

## The 15-turn budget

Consumer has up to 15 Q&A turns. Target: 4-8 core questions + some
follow-ups.

Strategy:

- **Round 1 (3-5 questions)**: top-priority items. These are your
  strongest leverage over the run.
- **Round 2 (2-3 questions)**: second-tier items, plus follow-ups
  generated by Round 1 answers.
- **Round 3 (rare, 1-2 questions)**: edge cases or ambiguities still
  unresolved.

Budget distribution guidance:

- If Round 1 answers are rich and clarify most things → skip Round 2 and
  3. The cap is a ceiling, not a target.
- If Round 1 answers generate many follow-ups → Round 2 handles those;
  may exhaust budget without Round 3.
- If you're in Round 3 still asking basic questions → the project type
  classification was probably wrong; reclassify or return
  `intake_blocked`.

---

## Batching within a round

One message per round, multiple questions within. Each question:

- Numbered
- Multiple choice when options are enumerable
- Includes a default for unanswered case
- Short enough to answer in one line each

Example Round 1:

> I need 3 clarifications before the rest of the framework can run:
>
> **1. Database**
> - (a) Postgres (recommended, Vercel Postgres free-tier)
> - (b) SQLite (simpler, harder to deploy)
> - (c) In-memory (toy only)
> Default: (a)
>
> **2. Authentication**
> - (a) Email+password with sessions
> - (b) Magic link via email
> - (c) Google OAuth only
> Default: (b) (lowest friction for users)
>
> **3. Expense categories**
> - (a) Fixed list I define (Food, Transport, etc.)
> - (b) User-defined categories
> Default: (a)

Note: every item has a default. Consumer never blocks waiting for an
answer — it proceeds on defaults if user stays silent.

## Question framing rules

### Always include options

Multiple choice > open-ended. Users answer multiple choice in seconds;
open-ended requires more thought and invites ambiguous answers.

Exception: when options aren't enumerable ("What's your app called?"),
ask directly but keep it bounded ("Name, under 30 chars?").

### Always include a default

Every question has a "default if not answered". This:
- Lets the user skip questions they don't care about
- Proceeds gracefully if they abandon the conversation
- Signals your own judgment about best choice

### Never ask for justification

Don't ask "why do you prefer X over Y?". User may not have a reason.
You don't need one. You need the choice.

### Don't ask for context they don't have

Don't ask "which transaction isolation level?" or "which vector
database?" unless the user has strongly signaled domain expertise. These
are implementation details; proxy them.

### Don't explain what the question means at length

A sentence of context is fine. A paragraph loses the user's attention.
Trust that if the user doesn't know, the default will serve them.

---

## Handling answers that generate new questions

Example:

> User: "Use Clerk for auth."
>
> New gaps:
> - Which Clerk plan / tier?
> - Which OAuth providers enabled in Clerk?
> - Where in the user journey does login appear?

These are Round 2 questions. Prioritize the same way — blast radius +
reversibility + ask-ability.

If the follow-ups are low-signal (user probably doesn't care), proxy
them instead of asking.

---

## When to refuse to ask

Sometimes a gap sits in "must-ask" per the taxonomy, but on reflection
the question is low-ask-ability (user can't reasonably answer). In that
case, reclassify as proxy-able:

- "What's your preferred testing strategy?" → proxy (default per
  decision-taxonomy)
- "How should we structure commits?" → proxy (convention)
- "What's the exact error format?" → proxy (convention; put in a
  convention spec)

Err toward proxying over asking for questions the user can't answer
quickly.

---

## Anti-patterns

- **Asking about things the user already said**: re-read the input
  before finalizing questions. Don't waste budget confirming what's
  covered.
- **Asking three questions in a row that should have been one**:
  "What framework? What routing? What state management?" — the first
  answer determines the rest. Ask framework, infer the others.
- **Leaving open-ended trailers**: "Anything else important?" — invites
  the user to dump unstructured info that's hard to process. End each
  round with a clean stopping point.
- **Asking after you have enough**: if must-ask items are all
  answered and proxy-able defaults fit, **stop asking**. Don't
  exhaust budget on nice-to-haves.
